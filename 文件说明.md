# VisionFlow 项目架构与解耦设计

## 一、四层架构总览

### 依赖方向（单向向下）

```
  gui/      展示层    ← 依赖 core.interfaces + services
  services/ 服务层    ← 依赖 core（无 Qt，无 GUI 引用）
  nodes/    算法层    ← 依赖 core.node_base + numpy
  core/     领域层    ← 依赖标准库 + numpy（零 GUI，零 Qt）
```

| 层级 | 目录 | 职责 | 依赖 | Qt 依赖 |
|---|---|---|---|---|
| **展示层** | `gui/` | 用户界面：面板、编辑器、控件 | `core.interfaces` + `services` | **有** (PyQt5) |
| **服务层** | `services/` | 桥接 core↔gui：DI 容器、生命周期、线程封装 | `core` | **无** |
| **算法层** | `nodes/` | 视觉处理算法：100+ 节点 | `core.node_base` + numpy | **无** |
| **领域层** | `core/` | 核心逻辑：ABC 接口、工作流引擎、事件总线、DI 容器 | 标准库 + numpy | **无** |

**关键约束**：gui 层**不直接修改** core 数据层状态，只读取和展示。GUI 通过 signals / callbacks / event_system 被动接收通知。

---

## 二、各层解耦机制详解

### 2.1 core/ 层 — 核心领域逻辑

#### 2.1.1 interfaces.py — ABC 接口契约

定义所有关键对象的抽象接口。GUI 和 services 依赖这些接口，不依赖具体实现类。

| 接口 | 用途 | 被谁依赖 |
|---|---|---|
| `INodeData` | 所有节点的最小契约 (node_id, name, ports, to_dict/from_dict) | gui, services |
| `IVisionNodeData` | 视觉处理节点 (message, result_image_source, invoke) | gui.result_panel |
| `ISrcFilesNodeData` | 文件数据源节点 (src_file_path, src_file_paths) | gui.flow_resource_panel |
| `IResultPresenterNodeData` | 可展示结果的节点 (result_presenter) | gui.result_panel |
| `IHelpNodeData` | 可展示帮助的节点 (help_presenter) | gui.help_panel |
| `IWorkflowEngine` | 工作流引擎 (start, stop, execute, state) | services.workflow_runner |
| `IProjectService` | 项目服务 (load, save, new_project) | gui.main_window |
| `INodeService` | 节点服务 (get_all_types, get_groups) | gui.toolbox_panel |
| `IEventBus` | 事件总线 (subscribe, publish, unsubscribe) | gui.log_panel |

**解耦关键**: GUI 组件使用 `isinstance(node, ISrcFilesNodeData)` 而非 `isinstance(node, SrcImageFilesNodeData)`。

#### 2.1.2 events.py — 事件总线 (EventSystem)

```
EventType 枚举: 25 个事件类型
  WORKFLOW_STARTED/COMPLETED/STOPPED/ERROR  — 流程生命周期
  NODE_STARTED/COMPLETED/ERROR               — 节点生命周期
  NODE_PROPERTY_CHANGED/NODE_SELECTED         — 节点交互
  PORT_STARTED/COMPLETED, LINK_STARTED/COMPLETED — 端口/连线生命周期
  DIAGRAM_CHANGED, NODE_ADDED/REMOVED          — 画布变更
  MESSAGE_INFO/WARN/ERROR/SUCCESS              — 日志消息
  PROJECT_LOADED/SAVED/CHANGED                 — 项目事件
  FILE_ITERATION_NEXT/COMPLETED                — 运行全部循环
```

**解耦机制**:
- 发布者只调用 `event_system.publish(EventType.XXX, sender=self, **kwargs)`，不知道谁在订阅
- 订阅者只注册回调 `event_system.subscribe(EventType.XXX, handler)`，不知道谁在发布
- Handler 签名统一: `handler(sender, **kwargs)`
- 全局实例 `event_system` 在模块加载时创建；`AppContext.init_defaults()` 会替换为新的实例（向后兼容过渡期）

**注意**: `AppContext.init_defaults()` 第 60 行 `_ev.event_system = self._event_bus` 会替换模块级 event_system 实例。由于 Python `from ... import` 捕获的是模块加载时的引用，已导入的模块持有旧实例引用。这是已知的过渡期限制，新代码应通过 `ctx.event_bus` 获取。

#### 2.1.3 ioc.py — DI 容器 (ServiceCollection / ServiceProvider)

对标 Microsoft.Extensions.DependencyInjection 的轻量实现。

**注册方式**:
```python
services = ServiceCollection()
services.add_singleton(IService, ServiceImpl)           # 单例
services.add_transient(IService, ServiceImpl)            # 每次新建
services.add_singleton(IService, instance=prebuilt)      # 预构建实例
services.add_singleton(IService, factory=lambda sp: ...) # 工厂函数
provider = services.build()
```

**构造器注入**: `ServiceProvider.resolve()` 通过 `inspect.signature` 检查构造函数参数，递归解析依赖。

**解耦关键**: GUI 组件通过构造函数接受服务接口（如 `ctx.project_service`），而不是 `import` 全局服务实例。

#### 2.1.4 node_base.py — 节点基类体系

```
NodeBase                        # 所有节点的根 (node_id, name, ports, to_dict/from_dict)
  └── HelpNodeDataBase          # + help_presenter
      └── DemoNodeDataBase      # + 示例参数
          └── VisionNodeData    # + Mat, invoke, _invoke_action, use_invoked_part
              └── ROINodeData   # + ROI 矩形管理
                  └── SrcFilesVisionNodeData  # + 文件列表管理, load_default()
                      ├── SrcImageFilesNodeData  (图像文件源)
                      ├── CameraCaptureNodeData  (摄像头源)
                      └── _ZooSrcImageBase       (Zoo 测试集)
```

**VisionNodeData 核心方法**:
- `invoke(previors, diagram)` → 查找源节点 → `_invoke_action()` → `invoke_core()`
- `_invoke_action(action)` → 发布事件 → `on_node_completed()` 写入历史 → 返回结果
- `update_invoke_current()` → 单步执行当前节点（用于实时预览）

**解耦关键**:
- `invoke_core(src_data, from_data, diagram)` 是抽象方法 — 节点开发者只需实现算法逻辑
- `_invoke_action` 自动处理事件发布 + 历史记录 + Mat 生命周期管理
- `diagram_data` 属性由 `WorkflowEngine.add_node()` 设置 — 节点不持有对工作流的硬引用
- `use_invoked_part` Property 控制是否参与历史记录（对标 WPF UseInvokedPart）

#### 2.1.5 workflow.py — 工作流引擎 (WorkflowEngine)

**核心职责**:
1. 节点图拓扑管理 (nodes + links)
2. 拓扑排序 + 串行/并行执行
3. 历史消息管理 (对标 WPF VisionDiagramDataBase.Messages)
4. 回调通知机制 (对标 WPF OnInvokedPart)

**历史消息系统 (完整解耦设计)**:

```
_invoke_action()                    [worker 线程]
  → workflow.on_node_completed()
    → 写入 VisionMessage 到 workflow.messages (加锁, 线程安全)
    → _notify_history_callbacks()
      → callback: _history_sync_requested.emit()  [worker 线程 → Qt 信号]
        → Qt::QueuedConnection marshaling 到主线程
          → sync_history_from_workflow()  [主线程, 安全操作 QTableWidget]
```

**关键方法**:
- `on_history_changed(callback)` — 注册回调（对标 WPF ObservableCollection.CollectionChanged）
- `on_node_completed(node, state, time_span)` — 记录节点执行结果
  - 对视频/摄像头节点: 按 `result_node_data` 匹配并**原地更新**已有行
  - 对其他节点: 新建 `VisionMessage` 追加
- `get_messages_snapshot()` — 返回线程安全的 Messages 副本
- `clear_messages()` — 清空历史（切换到新 diagram 时调用）

**解耦关键**:
- WorkflowEngine 是完全纯 Python 类（非 QObject），不依赖 Qt
- 通过回调机制通知 UI 层，而非直接操作 Qt 对象
- `_messages_lock` 保护 ThreadPoolExecutor 并发写入
- 回调在 worker 线程中执行 — 接收方（ResultPanel）通过 Qt::QueuedConnection 自动切回主线程

#### 2.1.6 result_presenter.py — 结果呈现数据模型

**纯数据 dataclass（零 GUI 依赖）**:

- `VisionMessage` — WPF VisionMessage 1:1 端口 (index, time_span, type_name, message, state, result_image_source, src_file_path, result_node_data)
- `NodeResult` — 完整节点执行结果 (node_id, success, message, value_items, rectangle_items, image_items...)
- `ResultItem` 系列 — 结果项基类 + RectangleResultItem / LineResultItem / ScoreRectangleResultItem / ImageResultItem / TableResultItem

**解耦关键**: 这些 dataclass 纯粹描述数据，不包含任何渲染逻辑。GUI 层读取后自行决定如何展示。

#### 2.1.7 project.py — 项目数据模型

- `DiagramData` — 单个流程图 (id, name, workflow, to_dict/from_dict)
- `ProjectItem` — 项目容器 (diagrams, selected_diagram_index, 序列化)
- `ProjectService` — 项目服务 (load, save, new, delete, 最近文件)

**解耦关键**: 项目 JSON 序列化通过 `to_dict()` / `from_dict()` 实现，不依赖 Qt 的 QSettings 或 QJsonDocument。

### 2.2 services/ 层 — 应用服务 (桥接层)

#### 2.2.1 app_context.py — 中央 DI 容器

替代所有全局单例的过渡方案。

**持有的服务**:
| 属性 | 类型 | 用途 |
|---|---|---|
| `event_bus` | `EventSystem` | 全局事件总线 |
| `node_registry` | `NodeRegistry` | 节点类型注册表 |
| `node_groups` | `NodeDataGroupBase` | 工具箱分组 |
| `project_service` | `IProjectService` | 项目 CRUD |
| `node_service` | `INodeService` | 节点查询 |
| `theme_service` | `IThemeService` | 主题管理 |

**关键方法**:
- `init_defaults()` — 创建默认事件总线/注册表/分组，并同步到模块级 global（向后兼容过渡期）

**解耦关键**: 新代码应通过 `ctx` 参数注入服务，而非 `import` 全局单例。`get_app_context()` 是过渡期的便捷入口，最终应全部改为构造器注入。

#### 2.2.2 workflow_runner.py — 流程运行器

封装工作流执行的线程管理和生命周期。

**执行模式**:
- `start_once()` — 单次执行（后台线程）
- `start_run_all(file_paths, auto_switch, interval)` — 遍历所有源文件（WPF "运行全部"）
- `start_continuous()` — 持续循环执行

**解耦关键**:
- Runner 使用标准 `threading.Thread`，不依赖 Qt 的 QThread
- 所有状态通知通过 `event_system.publish()` 发出（WORKFLOW_STARTED, FILE_ITERATION_NEXT 等）
- UI 通过订阅事件更新界面，Runner 不知道 UI 的存在
- `bind(workflow)` 绑定要执行的工作流 — Runner 是工作流的消费者，不是所有者

#### 2.2.3 node_service.py — 节点服务

封装 `NodeRegistry` + `NodeDataGroupBase`，提供查询接口给 GUI。

### 2.3 gui/ 层 — 用户界面

#### 2.3.1 GUI 如何从 core 层接收数据（读方向）

GUI 读取 core 层数据的方式，从高到低耦合：

| 方式 | 示例 | 耦合度 |
|---|---|---|
| **回调注册** | `workflow.on_history_changed(lambda: ...)` — ResultPanel 注册，workflow 消息变更时主动通知 | ★ 最松 |
| **EventSystem 订阅** | `event_system.subscribe(NODE_COMPLETED, handler)` — LogPanel, EditorWidget 订阅事件 | ★★ |
| **pyqtSignal 桥接** | `editor.node_executed.connect(main_window._on_node_executed)` — Editor 将事件转为 Qt 信号 | ★★★ |
| **直接属性读取** | `node.src_file_paths` — FlowResourcePanel 读取文件列表 | ★★★ |
| **反射/内省** | `Property` 描述符遍历 — PropertyPanel 动态生成表单 | ★★★ |

#### 2.3.2 GUI 如何向 core 层发送动作（写方向）

GUI 向 core 层发送用户操作的方式：

| 方式 | 示例 | 耦合度 |
|---|---|---|
| **方法调用** | `workflow.start()` / `workflow.add_node()` — 编辑器触发工作流操作 | ★ 最松 |
| **Setter 赋值** | `node.src_file_path = new_path` — FlowResourcePanel 切换选中文件 | ★★ |
| **EventSystem 发布** | `event_system.publish(NODE_SELECTED, sender=node)` — 节点选中事件 | ★★★ |
| **pyqtSignal** | `editor.node_selected.connect(main_window._select_node)` — 选中联动 | ★★★ |

#### 2.3.3 result_panel.py — 历史结果面板 (关键解耦案例)

这是最复杂的解耦案例，经过多轮迭代最终确定的设计：

```
历史数据流向 (写):
  node 执行 → _invoke_action → workflow.on_node_completed()
    → 写 VisionMessage 到 workflow.messages (worker 线程, 加锁)

历史数据显示 (读):
  workflow._notify_history_callbacks()
    → lambda: _history_sync_requested.emit()    [worker 线程 emit Qt 信号]
      → Qt::QueuedConnection marshaling          [自动切换到主线程]
        → sync_history_from_workflow()           [主线程]
          → get_messages_snapshot()              [线程安全读取]
            → 重建 QTableWidget                   [安全操作 Qt 控件]
```

**关键设计决策**:
- VisionMessage 是纯 dataclass（core 层），QTableWidgetItem 只在 GUI 层创建
- WorkflowEngine 不是 QObject，不依赖 Qt
- `_history_sync_requested = pyqtSignal()` 是 ResultPanel 内部信号，用于跨线程 marshaling
- `on_history_changed(callback)` 注册模式对标 WPF ObservableCollection.CollectionChanged
- 切换 diagram tab 时 → `bind_workflow(new_workflow)` → 注册新回调 → 旧回调仍在旧 workflow 上
- 点击历史行 → `msg.result_node_data` 直接引用节点 → 更新图像和属性面板

#### 2.3.4 flow_resource_panel.py — 图像源面板

**数据流**:
- 读: `node.src_file_paths` → 创建 ThumbnailButton 列表
- 写: 用户点击缩略图 → `node.src_file_path = path`
- 文件操作: `node.add_files()` / `node.add_files_from_folder()` / `node.delete_current_file()`
- 缩略图加载: `ThumbnailLoader(QThread)` 在后台线程加载，通过 `pyqtSignal` 发送 `QImage` 到主线程

**解耦关键**:
- `ThumbnailLoader` 发送 QImage（线程安全），不发送 QPixmap（Windows GDI 句柄线程绑定）
- `QImage(rgb.data, ...).copy()` 深拷贝 numpy 数据，避免悬空指针
- `add_files_from_folder()` 是节点方法，面板只是调用者

#### 2.3.5 property_panel.py — 属性编辑面板

**数据绑定机制**:
- 通过 Python `Property` 描述符反射获取节点的可编辑属性
- `Property(group=..., editor=..., choices=...)` 元数据驱动表单生成
- 修改属性时直接 `setattr(node, prop_name, new_value)` — 节点通知 PropertyChanged 事件

**解耦关键**:
- 属性元数据（display_name, group, description, readonly, choices, editor）全部在 core 层定义
- PropertyPanel 是纯消费者 — 不定义任何属性结构，只根据元数据渲染

#### 2.3.6 node_editor/ — 流程图编辑器

**Scene ↔ WorkflowEngine 双向绑定**:

```
Scene (GUI)                    WorkflowEngine (core)
──────────                     ────────────
add_node_item(node)    →       workflow.add_node(node)       [node.diagram_data = workflow]
remove_node(node_id)   →       workflow.remove_node(node_id)
add_link(link)         →       workflow.add_link(link)
remove_link(link_id)   →       workflow.remove_link(link_id)
```

**EditorWidget 事件桥接**:
```
event_system.publish(NODE_COMPLETED)  [worker 线程]
  → editor._on_node_completed()      [同步回调, worker 线程]
    → _state_queue.put()             [线程安全队列]
    → node_executed.emit()           [Qt 信号, QueuedConnection]
      → main_window._on_node_executed()  [主线程]
```

**解耦关键**:
- Scene 持有 `self._workflow` 引用 — 所有画布操作同步到 workflow
- EditorWidget 用 `_state_queue` + `QTimer(30ms)` 将 worker 线程的状态变更转移到主线程
- `node_executed` pyqtSignal 桥接事件系统到 Qt 信号体系
- `bind_workflow(workflow)` 方法支持动态切换工作流（tab 切换）

### 2.4 nodes/ 层 — 节点实现

所有节点遵循相同模式：

```python
class MyNode(OpenCVNodeDataBase, SomeBase):
    __group__ = "分组名"

    param = Property(default_value, name="参数名", group=PropertyGroupNames.RUN_PARAMETERS)

    def __init__(self):
        SomeBase.__init__(self)
        OpenCVNodeDataBase.__init__(self)
        self.name = "节点名称"

    def invoke_core(self, src_data, from_data, diagram) -> FlowableResult:
        # 纯算法逻辑
        mat = from_data.mat.copy()
        # ... 处理 ...
        return self.ok(mat, "处理成功")

    def is_valid(self, mat) -> bool:
        return mat is not None

    def _update_result_image_source(self):
        self._result_image_source = self._mat
```

**解耦关键**:
- 节点只实现 `invoke_core`（纯算法），不感知执行上下文
- `_invoke_action` 自动处理事件发布 + 历史记录 + Mat 生命周期
- Property 描述符自动提供属性编辑、序列化、变更通知
- 节点内不导入任何 GUI 模块

## 三、六大解耦模式总结

### 模式 1: ABC 接口隔离
```
gui 依赖 → core.interfaces.ISrcFilesNodeData  (抽象)
nodes 实现 → core.node_base.SrcFilesVisionNodeData  (具体)
```
GUI 不 `import` 具体节点类型，只检查 `isinstance(node, ISrcFilesNodeData)`。

### 模式 2: 回调注册 (Observer)
```
WorkflowEngine.on_history_changed(callback)
  → 数据变更时主动通知
  → 订阅者不知道其他订阅者存在
```
ResultPanel 不轮询，不依赖 EventSystem（已发现订阅被清空的问题）。

### 模式 3: Qt 信号跨线程 Marshaling
```
worker 线程 → pyqtSignal.emit() → Qt::QueuedConnection → 主线程 slot
```
WorkflowEngine 不依赖 Qt；Qt 信号只在 GUI 层内部使用，将数据从 worker 线程安全转移到主线程。

### 模式 4: Dataclass 数据传输对象
```
core.result_presenter.VisionMessage  (纯数据, 无 Qt 依赖)
  → gui.result_panel 创建 QTableWidgetItem 展示
```
数据传输对象（DTO）在 core 层定义，GUI 层负责渲染。

### 模式 5: 元数据驱动 UI
```
core.node_base.Property 描述符 (display_name, group, editor, choices...)
  → gui.property_panel 动态生成表单
```
属性结构在 core 层声明，GUI 层根据元数据自动渲染，无需为每个节点编写 UI 代码。

### 模式 6: 命令模式 (撤销/重做)
```
core.commands.Command (纯元组坐标, 无 Qt 依赖)
  → gui.node_editor.scene 调用 execute/undo
```
命令对象封装操作参数，不依赖 Qt 的 QUndoCommand。

```
  → 入度 = 0 的节点 → 第 0 层 (起始节点)
  → BFS:
    → 处理当前层节点的所有后继 (to_node_datas)
    → 后继入度减 1
    → 入度 = 0 → 加入下一层
  → 返回 [[level0], [level1], ...]

示例:
  [摄像头] → [灰度] → [阈值] → [轮廓检测]
                                    ↓
                              [Blob检测]

  拓扑排序结果: [[摄像头], [灰度], [阈值], [轮廓检测, Blob检测]]
  第 0-2 层: 串行执行 (每层 1 个节点)
  第 3 层: 并行执行 (2 个节点同时运行)
```

#### C5. 条件分支执行

```
ConditionNodeData.invoke_core()                                  [core/node_base.py:1150+]
  → 获取 from_node 的结果数据
  → 遍历 self.conditions (VisionPropertyCondition 列表):
    → cond.evaluate(from_node)                                   [core/node_base.py:1200+]
      → getattr(from_node, cond.property_name) — 读取上游属性值
      → 根据 cond.operator (">", "<", "==", "contains"...) 比较
      → 返回 True/False
    → 第一个匹配的 condition → 返回 Break 结果
      → FlowableResult.break_(mat, f"条件满足: {cond.output_node_id}")
      → 工作流引擎收到 Break → 只执行 output_node_id 对应的分支
    → 无匹配 → 走默认输出端口
```

**解耦要点**: `VisionPropertyCondition` 是纯数据类（core 层），`evaluate(node)` 通过 `getattr` 读取节点属性，不依赖具体节点类型。

1. **EventSystem 全局实例替换**: `AppContext.init_defaults()` 替换 `core.events.event_system` 会导致已导入模块持有旧实例引用。新代码通过 `ctx.event_bus` 获取正确的实例。历史结果功能已改用回调机制完全避开此问题。

4. **WorkflowEngine 回调重复注册**: `bind_workflow()` 每次调用都注册新回调，多次切换 tab 会累积。回调重复执行仅导致多余的表重建，不影响正确性。
